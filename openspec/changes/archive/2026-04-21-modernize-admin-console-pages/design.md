## Context

当前后台通过 [app.py](/Users/haojiayu/Documents/me/github/mtlogin-py/app.py) 的 `/dashboard` 路由一次性渲染 [templates/dashboard.html](/Users/haojiayu/Documents/me/github/mtlogin-py/templates/dashboard.html)，把平台配置、通知管理、账户管理、执行记录筛选、系统设置都放在同一个模板中。数据层已经完成平台、通知渠道、账户和执行记录的结构化建模，但页面组织仍停留在单页拼装阶段，导致导航、筛选、表单编辑和运行摘要彼此干扰。

这次变更不改动 Flask + SQLite 的技术栈，也不引入前后端分离，而是在现有服务端渲染模式上重构后台壳层、页面路由和模板组织。核心约束是继续复用当前 CRUD、调度器和认证流程，只调整 UI 结构与少量查询输出，以较低风险完成后台体验升级。

## Goals / Non-Goals

**Goals:**
- 将现有单页后台拆分为平台配置、通知管理、账户管理、执行记录查询、系统设置等独立页面。
- 建立统一的后台壳层，包括导航、页头、提示消息、统计摘要和响应式布局。
- 让账户管理列表展示最近一次执行返回的上传量、下载量、魔力值、最近一次登录成功时间，以及调度器计算出的下次执行时间。
- 保持现有表单提交流程和后端校验方式，避免为了页面重构而重写业务接口。
- 让系统设置从业务管理页中独立出来，继续承载管理员凭据修改能力。

**Non-Goals:**
- 本次不引入 SPA、前端构建工具或 REST API 重构。
- 本次不改变数据库实体关系，也不新增新的执行器或通知渠道类型。
- 本次不实现执行记录导出、批量操作或权限分级。

## Decisions

### 1. 采用“共享布局 + 独立页面模板”而不是继续扩展 `dashboard.html`

新增共享基础模板或局部模板片段，抽出侧边导航、顶部标题区、消息提示和基础样式。平台配置、通知管理、账户管理、执行记录查询、系统设置分别使用独立模板，只保留与本页职责相关的表格、筛选器和编辑表单。

原因：
- 当前单模板已经同时承担列表、编辑和筛选职责，继续扩展只会增加条件分支和样式耦合。
- 服务端渲染场景下，模板拆分是风险最小、收益最直接的重构方式。

备选方案：
- 继续保留一个 `dashboard.html`，通过锚点或折叠面板优化导航。放弃，因为页面职责没有真正拆分，复杂度仍然集中在一个模板中。

### 2. 使用独立路由承载每类管理能力，并保留根入口跳转

新增类似 `/accounts`、`/platforms`、`/notifications`、`/history`、`/settings` 的页面路由；现有 `/dashboard` 可以重定向到默认入口页，例如账户管理页。现有表单保存与启停接口保留原 POST 路由，提交后回跳到对应页面，而不是统一回到旧 dashboard。

原因：
- 独立 URL 是“独立页面”的必要条件，便于刷新、收藏和按页面维护状态。
- 继续复用 POST 接口和 flash 消息可以减少重构范围，避免牵动过多业务代码。

备选方案：
- 仅使用查询参数区分页面，例如 `/dashboard?page=accounts`。放弃，因为模板拆分和导航高亮都会更别扭，也不利于后续继续扩展。

### 3. 账户列表扩展为“基础配置 + 最近执行摘要 + 下次计划时间”

账户列表查询在现有最近一次执行记录关联基础上，补充输出上传量、下载量、魔力值、最近一次登录成功时间等字段，并在视图层增加 `next_run_at` 计算结果。`next_run_at` 不需要落库，可由调度器已有 cron 计算逻辑或复用同等规则在请求时推导。

原因：
- 用户明确要求在账户列表中直观看到运行产出与下一次执行时间，这些字段本质上来自最近执行记录和 cron 计划。
- 不新增持久化字段可以避免不必要的迁移，同时保持展示结果与最新调度状态一致。

备选方案：
- 在 `accounts` 表中冗余保存最近运行指标和下次执行时间。放弃，因为这些值天然属于派生数据，冗余后更容易失真。

### 4. 系统设置单独成页，但范围保持克制

系统设置页面首期只承载当前已经存在的管理员账号、密码修改等系统级配置，并复用现有保存逻辑与校验。该页面可以包含简短的运行环境说明，但不在本次扩展到日志查看、系统诊断或全局任务参数。

原因：
- 用户明确要求“系统设置”独立页面，但当前后端真正具备的系统级设置能力主要还是管理员凭据管理。
- 保持范围收敛可以确保这次变更聚焦在页面重构，而不是借机引入新的配置模型。

备选方案：
- 在系统设置页中同步加入日志、数据库维护、全局默认参数等新能力。放弃，因为这会把 UI 重构升级成新的产品功能开发。

### 5. 视觉升级采用统一设计令牌，不在每页各自堆样式

将颜色、字体、间距、卡片、表格、按钮等样式抽成共享设计令牌和基础组件类，页面层只组合布局，不重复声明完整样式。视觉方向以浅色现代控制台为主，强调信息分区、导航清晰度和移动端可读性。

原因：
- 当前内联样式全部写在单文件中，拆页后如果继续复制粘贴，会迅速失控。
- 统一令牌可以让新增页面保持一致风格，同时降低后续维护成本。

备选方案：
- 每个模板单独维护一份局部样式。放弃，因为这会造成样式漂移和重复劳动。

## Risks / Trade-offs

- [模板拆分后上下文变量不一致，导致表单或列表渲染缺字段] → 为每个页面定义明确的 `render_template` 上下文，避免继续把“全量后台数据”塞给所有页面。
- [账户列表的下次执行时间与调度器实时状态不完全一致] → 统一复用与调度器相同的 cron 计算规则，并对无效或空 cron 显示明确占位值。
- [保留原 POST 接口但切换回跳页面，可能遗漏某些 flash 跳转场景] → 为每类保存、启停、立即执行操作补齐来源页跳转逻辑，并在任务中加入端到端验证。
- [视觉重构后桌面端效果提升但移动端回退] → 共享布局优先按窄屏断点设计导航折叠与表格滚动，避免只面向大屏设计。

## Migration Plan

1. 新增共享后台布局模板、页面级模板和公共样式资源。
2. 将 `/dashboard` 拆分为多个独立 GET 页面路由，并调整现有 POST 动作完成后的回跳目标。
3. 扩展账户列表查询结果，注入最近执行摘要与下次计划执行时间。
4. 将管理员设置表单迁移到独立系统设置页面，并移除单页模板中的旧区块。
5. 更新 README 和截图/说明文案，确保入口路径与页面结构一致。

回滚策略：
- 如果拆页后出现严重问题，可让 `/dashboard` 重新渲染旧模板，并临时停用新页面路由。
- 数据模型不发生变化，因此回滚主要是模板和路由级别回退，不涉及数据库恢复。

## Open Questions

- 账户列表中的上传量、下载量、魔力值是否需要支持排序；本次先保证展示，不额外增加排序交互。
- 执行记录查询页是否需要分页；如果当前数据量不大可沿用现状，本次不把分页作为强制范围。
