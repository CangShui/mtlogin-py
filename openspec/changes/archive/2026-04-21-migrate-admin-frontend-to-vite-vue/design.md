## Context

当前后台前端直接由 Flask 渲染 HTML 模板，虽然已经拆成账户、平台、通知、执行记录和系统设置等独立页面，但页面交互、状态共享和后续扩展仍受限于服务器模板模式。用户希望前端改造为基于 Vue 的现代后台工程，并明确指定以 `/Users/haojiayu/Documents/me/github/v3-admin-vite-main` 作为实现基础。

被引用的模板项目已经提供后台布局、路由、Pinia、Element Plus、开发构建脚本和权限相关骨架，其当前 `package.json` 已定义了本次应沿用的依赖版本基线，其中 `vite` 为 `7.3.1`。因此这次设计不是“在模板基础上继续升级依赖”，而是需要在本项目内落地一个裁剪版管理台，并尽量保持与模板一致的技术栈版本，同时让 Flask 后端从表单式页面切换为 JSON API + SPA 资源分发模型。

## Goals / Non-Goals

**Goals:**
- 在本项目内引入基于模板同版本依赖集的前端管理台工程，结构参考 `v3-admin-vite-main`，首期基线为 `Vue 3 + Vite 7.3.1 + TypeScript + Element Plus`。
- 将当前后台管理能力迁移为 Vue 路由页面，包括登录、账户管理、平台配置、通知管理、执行记录和系统设置。
- 为前端提供受管理员会话保护的 `/api/admin/**` JSON API，并复用当前校验、存储和调度逻辑。
- 保持生产环境由 Flask 托管构建后的前端静态资源，同时支持本地开发阶段的前后端分离联调。
- 在迁移过程中尽量复用现有数据库模型与调度器行为，避免同步引入新的业务重构。

**Non-Goals:**
- 本次不引入新的业务实体、角色权限系统或多管理员模型。
- 本次不把后端改造成完全独立的 REST 服务，也不引入 JWT、OAuth 或前后端分仓。
- 本次不保留现有 Flask 页面模板作为长期双栈方案，服务器模板仅作为迁移过渡或回退手段。

## Decisions

### 1. 在仓库内新增独立 `frontend/` 工程，并以指定模板为骨架裁剪接入

本项目将新增一个独立前端目录，例如 `frontend/`，把 `/Users/haojiayu/Documents/me/github/v3-admin-vite-main` 的布局、路由、状态管理、样式体系和基础配置作为迁移基线，但只保留与本项目后台场景有关的模块。演示页、权限演示、表格示例等无关内容会在接入时删除。

原因：
- 用户已经指定具体模板项目，希望直接基于该工程风格与结构实现。
- 独立目录比继续把资源塞进 Flask `templates/` / `static/` 更利于构建、类型检查和前端依赖管理。

备选方案：
- 继续在现有 Flask 模板体系中渐进式引入 Vue 组件。放弃，因为最终仍会保留双重渲染模型，复杂度更高。

### 2. 使用 Flask Session Cookie 继续做管理员认证，不额外引入 JWT

前端登录页改为 Vue 页面，但登录态仍由 Flask 管理。后端提供会话相关接口，如登录、登出、当前会话状态查询；前端通过 `fetch/axios` 携带同源 Cookie 完成鉴权。所有 `/api/admin/**` 接口继续沿用 `login_required` 一类的会话校验思路，未登录时返回 JSON 401，而不是 HTML 重定向。

原因：
- 当前后端已经具备稳定的 session 认证逻辑，迁移时保留这一点风险最低。
- 本项目是单体应用，同源前后端场景下没有必要为了 SPA 额外引入 token 体系。

备选方案：
- 切换为 JWT 或双 token 模型。放弃，因为这会把一次前端重构扩展成认证系统改造。

### 3. 新增 `/api/admin/**` JSON API，页面表单提交改为异步请求

现有平台、通知、账户、执行记录和系统设置的页面读写逻辑会被拆成 JSON API，包括列表查询、详情读取、保存、启停、立即执行、历史筛选等。原本散落在服务器模板中的 flash 消息、编辑态参数和跳转逻辑，前端将改为本地路由状态、对话框或消息提示处理。

原因：
- Vue 单页应用不能继续依赖模板表单提交和 HTML 重定向作为主要交互模式。
- JSON API 可以最大化复用现有 `SettingsStore`、`BackgroundScheduler` 和校验逻辑。

备选方案：
- 仅对部分页面做 JSON 化，保留部分表单 POST。放弃，因为最终会形成混合交互模型，前端状态流难以统一。

### 4. 生产环境由 Flask 托管构建后的 SPA，并为前端路由提供回退

Vite 构建产物输出到后端可访问目录，Flask 在生产环境分发 `index.html` 和静态资源，对 `/accounts`、`/platforms`、`/notifications`、`/history`、`/settings` 等前端路由统一回退到 SPA 入口；而 `/api/admin/**` 保留为真实后端接口路径。

原因：
- 用户当前项目仍是单仓库、单服务形态，最简单的部署方式是由 Flask 一并托管最终产物。
- 这样不需要额外引入 Nginx 配置或双服务编排，部署心智负担最低。

备选方案：
- 让前端独立部署，后端只暴露 API。放弃，因为这会显著增加部署和跨域配置复杂度。

### 5. 模板项目按“同版本迁移”方式接入，而不是先升级前端基础设施

本变更会优先沿用 `/Users/haojiayu/Documents/me/github/v3-admin-vite-main` 中已经验证过的技术栈版本，包括 `vite 7.3.1` 及其相关插件版本，并在本项目中完成裁剪接入。只有在本项目实际接入时遇到明确兼容性问题，才局部调整依赖，而不把“升级构建栈”作为本次默认范围。

原因：
- 用户已经明确改为“使用模板里的技术栈版本”。
- 优先保持模板版本一致，能降低接入风险，也能减少迁移过程中的插件兼容变量。

备选方案：
- 在迁移过程中同步升级到更高版本的 Vite。放弃，因为这会把一次模板接入扩大成基础设施升级任务。

## Risks / Trade-offs

- [模板项目原始依赖在本项目内接入时出现兼容问题] → 先最小化保留必要插件与布局能力，仅在确有必要时局部调整依赖，而不是默认整体升级技术栈。
- [SPA 接入后后端 API 与前端页面迁移节奏不一致] → 优先先完成认证、导航和核心 CRUD API，再逐页迁移视图，保持每个页面可独立联调。
- [现有 Flask 页面与 SPA 路由共存时出现路径冲突] → 统一把数据接口放在 `/api/admin/**`，前端页面保留在根路径下的管理路由，避免同路径混用。
- [部署环境缺少 Node/pnpm 构建步骤] → 在 README 和任务中明确前端构建命令、输出目录与生产部署流程。

## Migration Plan

1. 在仓库中创建 `frontend/` 工程，并从指定模板迁入最小后台骨架。
2. 先补齐登录会话、导航和核心管理数据的 `/api/admin/**` 接口。
3. 逐步把账户、平台、通知、执行记录、系统设置页面迁移为 Vue 路由页面。
4. 配置 Vite 开发代理与生产构建输出，使 Flask 可以同时提供 API 和 SPA 入口。
5. 完成页面联调、构建验证和文档更新后，再清理旧模板主流程。

回滚策略：
- 若迁移期间出现重大问题，可以暂时保留现有 Flask 模板入口，停用前端 SPA 路由入口。
- 数据模型与调度逻辑不改变，因此回滚主要集中在路由分发和页面入口层。

## Open Questions

- 前端工程目录最终命名为 `frontend/` 还是 `admin-frontend/`；本设计默认使用 `frontend/`。
- 登录页是否完全沿用模板视觉风格，还是保留当前项目品牌信息；本次先按模板结构接入，再做项目化定制。
- 是否保留少量 Flask 模板作为应急回退入口；本设计允许短期保留，但不把它作为长期目标。
